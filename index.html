T.E.A.M. Carpool IOS App

Together Everyone Achieves More.

  • Date: December 2014
  • Client: Interaction Design @ UT Austin
  • Category: Mobile Application Prototype
  • Team: Luping Yeh, Ran Xu, Zhen Wang
  • Role: User Research, Interaction Design

T.E.A.M. is the location-based mobile social network application for ridesharing, ride combining, and carpooling on a mobile platform, which aims to conserve resources to lessen environmental impacts while also connecting people.

 T.E.A.M. as an activity can save you money besides getting you more socialize and meet new people. It allows user to share the commute with someone else or several someone, which means you can share the cost of gas as well. T.E.A.M. also creates a way to find ride pools for your daily commute in and around your city, to share memories along the road trip, or to ease frustration of finding parking space in hot travel spots and festivals. Best of all, you get to use the carpool lane!

Executive Summary

Based on the 2014 Parking Strategies Committee Report of University of Texas at Austin, UT had 52,000 students, 17,279 faculty and staff, 5,000 visitors and 15,800 parking spaces in the year of 2013, which means the potential user to parking space ratio is 4.8:1. In short words, UT has a severe parking situation. To solve the problem, UT now allows a max of four members to share a parking permit and carpool to the campus. However, based on our user research, few students know the existence of this policy and the major problem for them to take advantage of this policy, is to find a carpooler.

Based on the reasons above, we designed a carpool app called T.E.A.M. for the UT students to find a carpooler. For one thing, we can help students find a carpooler quickly and much more easily. For another thing, the app advertised UT’s carpool permit policy, which can raise UT’s profit. In general, the app help with the severe parking situation for UT, and help students save time and money on their way to school.

In the final implementation of the T.E.A.M. Carpool app, we accomplished five major functions together with all other related features. Users will be able to search for a carpooler and use the filter to limit the search results. If the user has not found a satisfied ride, he or she can post a ride to look for a carpooler. Drivers and passengers can communicate through messages, and both of the two groups can add the satisfied ride to favorite for later review. In user account page, users can check his or her own profile, carpool history, and information about UT carpool permit. In general, all users can complete the general tasks of search, find satisfied carpooler, communicate with the carpooler, reserve the ride and confirm the end of the ride.

As a team of three people, we completed the whole process of mobile app designing: from the brainstorming of initial concepts, competitor analysis, stakeholder and user research to low fidelity and high fidelity prototype. User tests meant a lot in our project because each round of user test led us to a new version of prototype. The whole process gave us a brand new understanding of interaction design, and help us digest what we learned from the class.

User Research

From the interview results, we summarized the user needs

  • With-car student:
  1. User should be able to find a carpooler whenever they need.
  2. User should be able to find a carpooler living in the same area.
  3. User should be able to find a carpooler with similar schedules.
  4. User should be able to get a discount for parking permit if they carpool.
  5. User should be able to get free parking in garage if they carpool.
  6. User should be able to get shared gas fee
  • Without-car student:
  1. User should be able to check drivers’ class schedule
  2. User should be able to track where the car is
  3. User should be able to see if the driver is online or available
  4. User should be able to connect drivers’ facebook, and schedule appointment
  5. User should be able to share gas fee with or give rewards to the driver easily
  6. User should be able to find nearby users
  7. User should be able to see and contact drivers easily

Innovation Lab Team Site

Simplify the Project Assignment Workflow

  • Date: May 2014
  • Client: University of Nebraska-Lincoln
  • Category: Web Design & Development
  • Team: Dalton Dick, Zhen Wang
  • Role: Web Design & Development, Database Desgin & Development

This project is to develop a website using Ruby on Rails. The purpose of the website is to provide a platform for cse innovation lab (cse senior design) to better assign students projects in a convenient way and project team cooperation. It allows students to choose projects of interest, and sponsors to give preference by overseeing students’ application which includes students’ resume and cover letter. Based on the preference from student and sponsor, instructors will assign projects to each student. Once students are assigned, team will be formed for them as well. Team members can work together through the website.

The team site has greatly simplified the project assignment workflow and is therefore able to save a substantial amount time and effort for the course instructors. The system is an integral component of the Innovation Lab (iLab)4 infrastructure, a life cycle management facility for the capstone course. Its capabilities include a web portal for handling student and sponsor registration and data entry, project information exchange and team progress management in addition to project assignment. At the time of this writing, the iLab infrastructure project is still under active development. Nevertheless, we have agreed upon the programming interfaces that will be used for future integration.

Citportal Website

The Combinational Interaction Testing(CIT) Web Portal.

  • Date: September 2013
  • Client: University of Nebraska-Lincoln
  • Category: Web Design & Development
  • Team: Zhen Wang
  • Role: Web Design & Development, Database Design & Development

This website is called citportal. I developed it, using PHP with codeigniter. The site is to serve as a resource for combinatorial interaction testing in many domains. The website is in its early development stages, more new contents will be added,such as educational/research resources. Click here to browse the website.

This website had an original version, but it was a static page, no database connected. In order to make it easier to edit and add new stuff to the website, I decided to re-implement it as dynamic website. So I discussed with sponsor to settle down the requirements and determine the entities we need for the site. Then I started building the database, using mysql and PhpMyAdmin (UI based database management tool).

In order to make the site look better, I used the grumby framework to help build the website. With the framework, it is easier to set the grid, and arrange contents. As this is an informative site, I organized the contents as pieces of information boards, and used light but high contrast colors basically black, white, gray and some other highlights colors, to make the contents more clear and well-arranged to users.

iBulletin Database Development

iBulletin Connects iSchoolers.

  • Date: December 2014
  • Client: Database Management @ UT Austin
  • Category: Web & Database Development
  • Team: Cara Polisini, Zhen Wang
  • Role: Database Design & Development, Web Desgin & Development

iBulletin is an online bulletin board designed to help connect the iSchool community outside of classes. Although the iSchool has the luxury of being housed in one building, it can be difficult to meet people outside of class. The iSchool building does not have the sort of space that allows large numbers of students to study and hang out once their classes are finished. Additionally, many iSchool students work full time and are not able to participate in student groups or attend lunchtime lectures. iBulletin is a virtual space that will allow iSchoolers to connect with other members of the iSchool community who they might not otherwise meet. iBulletin will enable iSchool students to share interests, events, and opportunities in Austin. The social networking site Meetup.com inspired iBulletin. Meetup.com facilitates new friendships by connecting users through mutual interests, and many people join Meetup.com when they move to a new city. Once a group plans an event on Meetup.com, other users can see the post and join. We want iBulletin to have a similar environment and connect iSchoolers through mutual interests. iBulletin is not just the iSchool version of Meetup.com, it also incorporates aspects of Facebook’s design.

iBulletin combines attributes of Meetup.com and Facebook to create an online environment that allows users to plan real world events, but also to interact solely online. iBulletin is an organized bulletin board where students can post articles, threads, comments, and links by topical category. Some of those posts may result in real world meet ups, but some may just generate an online conversation. The ultimate goal is for more iSchool students to meet other students with similar interests, and to be aware of events and opportunities in Austin. This website will give students another way to meet other iSchoolers, outside of class and school events.

DB Descriptin (Entities)

User Entity: the user table is the center of the E-R model, to store all the users’ login information, including the user name and password. In most cases, only login user can take actions on iBulletin site. In other words, the user ID can trace almost any action taken within iBulletin. And we use user_type to distinguish admin (user_type is 0) and normal user (user_type is 1). Profile is separate from user in order to split up the amount of data that needs to be kept in one table. Once inside the site the user’s profile information can be accessed.

Profile Entity: the profile table records the contact information for the user. The Profile table also stores the user’s avatar, which gives the user their visual online presence. Profile is only linked to user because it acts as a repository for information about the user, and we wanted to break up the amount of information stored in one table.

Category Entity: the category table stores the topics that are posted about in iBulletin where we can track user’s interest by looking at the topics included in the posts. The purpose of the category is to help organize the topics of posts that are posted on iBulletin. Category is connected to the Post table (many-to-many relationship).

Post_has_category Entity: the post_has_category table stores the category id and post is whisch is used to connect the post table and category table. It is used to help track which the category or categories the post belongs to.

Post Entitiy: the post table stores the information that users will read. The posts are written by users and include a title and a body of content (post_description). The post content is the bulk of the material, but users can also add a photo or bookmark to enhance the post which is store in photo table and bookmark table respectively. The posts are sorted into categories, so users can search for posts based solely on their interests.

Bookmark Entity: the bookmark table records the external links that a user might put in a post. Users might want to link to a site outside of the database and the bookmark table stores that URL. The Bookmark table is linked to the Post table (one-to-many relationship). A bookmark can be traced using the post_id.

Photo Entity: the photo table holds all of the metadata about the photo included in a post. Instead of the real photo metadata, we store the file path and file name of the photo, which can help generate the URL for the photo. The photo was separated from the post table, in order to limit the amount of information stored in one table and also enable user to store multiple photos for a post. The photo table is only connected to the post table (many-to-one relationship) because it is primarily meant as a supporting feature to the post.

Post_rating Entity: the post_rating table stores the information of the comments from users for each posts. It allows users to express their opinions about a post. The concept is very similar to the ‘like’ button with comment box on Facebook. Users are able to quickly show their support without making any extra effort. If the user wants to leave a comment, then the comment is also stored in the post_rating table. Post_id and user_id are both able to track the ratings generated.

Feedback Entity: the feedback table is meant for communication between the administrators and the users. This is private communication for questions, complaints, site problems etc. All that is stored is the user_id of the user who writes the comment, the content of the comment, and the dates associated with the message.
